# 单用户私用同步测试指南

项目当前采用单用户私用模式，不提供注册、登录、退出登录或多账号隔离。本文档保留原文件名，作为从多账号方案收敛后的同步与离线测试清单。

## 测试前提

- 使用本地静态服务器或 GitHub Pages 访问应用。
- Supabase 表中存在 `user_id = '87553652@qq.com'` 的私有数据行。
- 浏览器支持 IndexedDB、Service Worker 和 BroadcastChannel。

## 启动与降级

1. 清空站点数据后打开应用。
2. 预期：无需登录，页面进入主界面。
3. 正常联网时，应用从 Supabase 拉取单用户数据。
4. Supabase 或 CDN 不可用时，应用应在短时间内进入本地模式，不应长时间停留在“同步数据中...”。
5. 离线模式下，添加动作和打卡仍应写入 IndexedDB。
6. 顶部同步状态应显示“本地模式”“待同步”或“离线，本地已保存”。

## 本地数据保存

1. 进入“胸”部位，添加动作“杠铃卧推”，组数 4，次数 10，重量 60kg。
2. 点亮 2 个打卡圆圈。
3. 刷新页面。
4. 预期：动作和已完成的 2 组仍保留。

## 跨标签页同步

1. 同一浏览器打开两个应用标签页。
2. 在标签页 A 添加动作或打卡。
3. 预期：标签页 B 通过 BroadcastChannel 或 localStorage pulse 同步状态。

## 云端同步

1. 在联网状态添加或修改动作。
2. 等待 1 秒后刷新页面，或在另一个设备打开应用。
3. 预期：Supabase 中同一 `user_id` 的数据被更新，另一端能看到最新状态。
4. 若 Supabase 已存在同一 `user_id` 的数据，恢复联网或刷新后应以远程数据为准覆盖本地缓存；只有 Supabase 行不存在时才通过 `upsert` 自动创建。

## 历史与隔夜重置

1. 完成任意训练组。
2. 调整测试环境时间跨过凌晨 2:00 后刷新，或通过浏览器调试模拟时间。
3. 预期：前一天完成记录进入历史日历，今日打卡状态重置。
